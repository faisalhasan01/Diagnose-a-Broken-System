How this Performance Optimization was Diagnosed & Resolved

In Slow (Unoptimized) Mode, the API suffers from a critical N+1 Query Regression. For each order fetched, Django executes individual database lookup queries to load the related Customer, the Payment details, and the items inside the order, followed by additional queries to load the Product details for each item. This results in hundreds of queries for standard payloads.

Unoptimized N+1 Query Sequence

# Triggered once for list of orders:
SELECT * FROM orders_order;

# Triggered N times (once per order):
SELECT * FROM orders_customer WHERE id = X;
SELECT * FROM orders_payment WHERE order_id = Y;
SELECT * FROM orders_orderitem WHERE order_id = Z;

# Triggered N * M times (once per item):
SELECT * FROM orders_product WHERE id = W;

Optimized Query Sequence (Exactly 2)

# Query 1: Join orders, customer, and payment
SELECT * FROM orders_order 
INNER JOIN orders_customer ON ... 
LEFT OUTER JOIN orders_payment ON ...;

# Query 2: Fetch related items and products in bulk
SELECT * FROM orders_orderitem 
INNER JOIN orders_product ON ... 
WHERE order_id IN (...);

By utilizing Django's select_related() for ForeignKey/OneToOne joins and a prefetch_related() with a custom Prefetch() query set for the many-to-many items, we consolidated the database trips down to exactly 2 database queries, regardless of the size of the dataset.
